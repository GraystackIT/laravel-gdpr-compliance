# Laravel GDPR Compliance Package — Implementation Plan

> Living document. Source of truth for the implementation. Updated as decisions are made.

---

## 1. Goal

Build a publishable Composer package `graystackit/laravel-gdpr-compliance` that gives any Laravel 13 application a complete GDPR/DSGVO compliance toolkit:

- Personal data declaration on Eloquent models via a fluent DSL
- Subject data export (Art. 15)
- Subject data erasure with grace period and optional legal hold (Art. 17)
- Per‑purpose consent management (Art. 6, 7)
- Cookie consent helper
- Audit log for compliance evidence (Art. 30)
- Policy version tracking
- Composer/NPM package inventory snapshots
- Async processing via queue jobs
- Configurable middleware for consent‑gated routes
- Eight Artisan commands for operations

The host repo at `/Users/bharb/Development/Packages/laravel-gdpr-compliance` is currently a Laravel 13 app skeleton. It will be restructured into a Composer library, then published to Packagist as `graystackit/laravel-gdpr-compliance`.

## 2. Locked Decisions

| # | Decision |
|---|---|
| 1 | **Vendor / namespace**: `graystackit/laravel-gdpr-compliance` with PSR‑4 `GraystackIt\Gdpr\` → `src/` |
| 2 | **No Spatie ActivityLog dependency.** Spatie's package is for user‑facing activity ("User XY ordered item Z"). Our `gdpr_audits` is a separate, technical, legally‑required audit log. |
| 3 | **`PersonalData` contract uses a fluent DSL.** Method receives a `PersonalDataBlueprint` builder, configures it, returns it. Full IDE autocomplete, refactor safety, enums. The blueprint is both builder and immutable value object (frozen on `build()`). |
| 4 | **Profile resolution is non‑merging**: model implementing `PersonalData::personalData()` wins entirely. Config `models[$class]['profile']` is fallback only for vendor models. Empty default if neither is present. |
| 5 | **No closures in config**: all FQCNs as strings; config stays cacheable. |
| 6 | **Long‑running subject operations are async via queue jobs.** Export, anonymization, deletion. Inventory scan is **synchronous** — it's a dev tool that runs once on `composer update`, not user‑facing. Tests use `Bus::fake()` for the async jobs. |
| 7 | **The package never touches Eloquent lifecycle traits on host models.** No forced `SoftDeletes`, no observers, no global scopes. The grace period state lives entirely in `gdpr_deletions`. The host model row is left untouched until the grace period expires. |
| 8 | **`saveQuietly()` for anonymization** to prevent observer loops; audit is logged explicitly via `AuditLogger`. |
| 9 | **Trait split**: `HasPersonalData` (any model with PII; passive) + `IsPersonalDataSubject` (initiator: `User`, `Customer`, …; can call `requestDeletion`, `requestExport`). Only subjects can initiate. |
| 10 | **Package inventory is a JSON file**, not a DB table. Generated by `gdpr:packages-scan` command. Helper `GDPR::packageInventory()` exposes the data. The user wires the command into their own host app's `composer.json` `post-update-cmd`. |
| 11 | **`loadMigrationsFrom` is NOT called in the ServiceProvider**. Migrations must be published. Tests load them via `TestCase::defineDatabaseMigrations()`. |
| 12 | **Workbench model location**: `workbench/app/Models/` (Orchestra Testbench convention). |
| 13 | **IDE / tooling files** (`.idea/`, `.vscode/`, `.nova/`, `.zed/`, `.fleet/`, `boost.json`, `.mcp.json`, `.ai/`, `AGENTS.md`) are kept locally and added to `.gitignore`. They are not part of the published package. `AGENTS.md` is required by Laravel Boost. |
| 14 | **Retention modes**: `delete` \| `anonymize` \| `legal_hold`, configured per model. |
| 15 | **Grace period**: optional, default `0` (no grace, immediate processing). Hard cap at 30 days. `PersonalDataBlueprint` throws on construction if higher (DSGVO Art. 12(3)). The host app must opt into grace explicitly per profile. |
| 16 | **Audit log contains no PII.** Event‑driven, never field‑change‑driven. Logged events: `deletion_requested`, `deletion_scheduled` (per affected model), `deletion_cancelled`, `anonymization_completed`, `deletion_completed`, `legal_hold_started`, `legal_hold_expired`, `export_requested`, `export_completed`. Consent and policy events are NOT in `gdpr_audits` — they live in their own append‑only tables (see Decision #25). Per entry: subject FK (orphans after delete are intentional), `target_model` for per‑model events, affected row count, affected field names (never values). No values, no raw user agent, IPs only as `xxx.xxx.xxx.0` if at all. Pruned per Decision #24. |
| 17 | **Domain events** are fired for external integration: `PersonalDataDeletionRequested`, `PersonalDataDeletionCancelled`, `PersonalDataAnonymized`, `PersonalDataErased`, `LegalHoldStarted`, `LegalHoldExpired`, `PersonalDataExported`. |
| 18 | **Registry‑driven processing.** `config('gdpr.models')` is the single source of truth for which models the package processes. Subjects do **not** declare relations — every PII‑holding model is in the registry, and each model resolves its own subject scope via `scopePersonalDataForSubject(Builder $query, Model $subject)`. No graph walking, no cycle detection, no depth limit. |
| 19 | **Profile schema is minimal and opt‑in.** Per field, both behaviors are opt‑in: `anonymizeWith()` to mark for anonymization, `exportable()` to mark for export. A field can be one, the other, both, or — illegally — neither (throws on `build()`, the field would be functionless). No `loggable`, no `redact_in_log` — the audit log is event‑driven and never sees field values, so per‑field log flags are meaningless. |
| 20 | **Subject↔Subject is hands‑off.** When a Subject is processed, other Subjects are never modified. Foreign keys to deleted subjects must be handled by the host app via DB constraints (`onDelete('set null')`) or event listeners on `PersonalDataErased`. |
| 21 | **`gdpr_deletions` is one row per (subject × affected model).** When `requestDeletion()` is called on a subject, the scheduler walks the registry and creates one `gdpr_deletions` row per registered model that has rows for this subject — plus one row for the subject itself. Each row carries its own `retention_snapshot` (JSON) capturing what the profile said at request time. |
| 22 | **Snapshot wins over live profile at processing time.** Cron Pass 1 reads `retention_snapshot` from `gdpr_deletions`, not the live model profile. Profile changes between request time and processing time do not retroactively alter scheduled work. |
| 23 | **Auth integration is the host app's responsibility.** During grace, the subject row is unchanged and a determined user can still log in. The package exposes `GDPR::isDeletionPending($subject)` and a `RequireNoDeletionPending` middleware. The host app wires these into auth/login flows according to its own UX policy (Google‑style "log in to cancel" vs. Facebook‑style "locked out"). |
| 24 | **`consents`, `gdpr_policy_acceptances`, `gdpr_audits` and `gdpr_requests` are append‑only and time‑pruned.** Default retention is **3 years** (per § 195 BGB / § 1489 ABGB and DPA best practice). Pruned via `gdpr:prune` command, scheduled by the host app. `consents` prune preserves the latest row per `(subject, purpose)` as the current state. `gdpr_policy_versions` (the version definitions themselves) are never pruned. `gdpr_requests.notification_email` is wiped separately after 7 days in terminal state (see Decision #27). |
| 25 | **`AuditLogger` is exclusively for the deletion/export pipeline.** Consent grants/withdrawals are recorded only in the `consents` table (append‑only). Policy acceptances are recorded only in `gdpr_policy_acceptances` (append‑only). No double logging into `gdpr_audits`. |
| 26 | **Processing order is declared per profile via `->processOrder(int)`.** Lower numbers process first, the subject typically uses `1000` (last). Default `100`. The `DeletionScheduler` sorts `gdpr_deletions` rows for a request by `process_order` and dispatches them as a `Bus::chain()` so they run sequentially in deterministic order. This solves FK constraint violations when host models have NOT NULL foreign keys without cascade. The order is snapshotted into `gdpr_deletions.process_order` at request time (per Decision #22). |
| 27 | **Notifications are sent via Laravel Notifications with email snapshotting and two‑layer cleanup.** The package ships four default notifications (`DeletionRequested`, `DeletionCancelled`, `DeletionCompleted`, `ExportReady`), overridable per class via `config('gdpr.notifications')` (null = use default, false = skip, FQCN = user class). Translation strings in `lang/en/gdpr.php` and Blade views are publishable for branding. At request time, the subject's current email is snapshotted into `gdpr_requests.notification_email` so it survives subsequent anonymization/deletion. **Email lifecycle**: Layer 1 — the email is wiped in the same transaction as the final notification send (on `DeletionCompleted`, `DeletionCancelled`, `ExportReady`). Layer 2 — `gdpr:prune` wipes any remaining emails on terminal requests older than 7 days (backup for failed notifications). The `gdpr_requests` row itself is kept for 3 years as compliance proof, without PII. |

## 3. Architecture Overview

### Layer Responsibilities

| Layer | Responsibility |
|---|---|
| **Contracts** | Interfaces: `PersonalData`, `Anonymizer` |
| **Enums** | `ConsentPurpose`, `RetentionMode`, `DeletionState`, `RequestType`, `RequestStatus` |
| **Models** | Eloquent models for the 5 GDPR tables |
| **Support** | Business logic services (`GdprManager`, `Exporter`, `Eraser`, `DeletionScheduler`, …) |
| **Traits** | `HasPersonalData`, `IsPersonalDataSubject`, `HasConsentRecords` |
| **Anonymizers** | 7 concrete anonymizer implementations |
| **Middleware** | `RequireConsent`, `ApplyConsentCookies`, `RequireNoDeletionPending` |
| **Commands** | 8 Artisan commands |
| **Jobs** | 3 queued jobs |
| **Events** | 7 domain events for external integration |
| **Facade** | `GDPR` facade proxying `GdprManager` |

### Database Tables (5)

| Table | Purpose |
|---|---|
| `consents` | Per‑purpose consent records, morphs to subject |
| `gdpr_requests` | Top‑level request lifecycle (export / anonymize / delete), one row per user‑facing request |
| `gdpr_deletions` | Scheduled deletion work, one row per (request × affected model). Holds per‑row retention snapshot, state, scheduled_for, hold_until |
| `gdpr_audits` | GDPR event audit log (technical, no PII). Per entry: subject FK, event, target_model (nullable), affected_rows (nullable), context JSON |
| `gdpr_policy_versions` | Policy version records and subject acceptance records |

> Note: `gdpr_package_inventories` is not a table. Inventory is a JSON file.

### Processing Model

The package processes subjects via a **flat registry** of all PII‑holding models. There is no relation graph to walk:

```
$user->requestDeletion()
       │
       ▼
DeletionScheduler iterates config('gdpr.models')
       │
       ├─ User::class           → finds subject row, plans User-mode processing
       ├─ Order::class          → asks Order::scopePersonalDataForSubject($q, $user)
       │                         → finds 12 rows
       │                         → plans Order-mode processing (legal_hold 10y)
       ├─ Comment::class        → scope returns 4 commenter rows → plans anonymize
       ├─ Address::class        → scope returns 1 row → plans delete
       └─ Vendor::class         → scope returns 0 rows → SKIPPED, no row created
```

Each registered model is **self‑describing** via two pieces:

1. Its **profile** (declared in `personalData(PersonalDataBlueprint $b)`)
2. Its **subject scope** (declared in `scopePersonalDataForSubject(Builder $q, Model $subject)`)

The same iteration is used for export, with the exporter collecting rather than wiping.

## 4. Profile DSL

### Subject model — `User`

```php
namespace App\Models;

use Illuminate\Foundation\Auth\User as Authenticatable;
use GraystackIt\Gdpr\Contracts\PersonalData;
use GraystackIt\Gdpr\Enums\RetentionMode;
use GraystackIt\Gdpr\Support\PersonalDataBlueprint;
use GraystackIt\Gdpr\Traits\HasPersonalData;
use GraystackIt\Gdpr\Traits\IsPersonalDataSubject;
use GraystackIt\Gdpr\Traits\HasConsentRecords;

class User extends Authenticatable implements PersonalData
{
    use HasPersonalData, IsPersonalDataSubject, HasConsentRecords;

    public function personalData(PersonalDataBlueprint $b): PersonalDataBlueprint
    {
        return $b
            // PII: anonymize AND export
            ->field('name')
                ->anonymizeWith('name')
                ->exportable()
            ->field('email')
                ->anonymizeWith('email')
                ->exportable()
            ->field('phone')
                ->anonymizeWith('phone')
                ->exportable()
            ->field('last_login_ip')
                ->anonymizeWith('ip_address', ['mask' => 'octet'])
                ->exportable()
            ->field('bio')
                ->anonymizeWith('free_text', [
                    'replace_email' => true,
                    'replace_phone' => true,
                    'replace_urls'  => true,
                ])
                ->exportable()

            // PII internal: anonymize only, do NOT export
            ->field('password')
                ->anonymizeWith('static_text', ['value' => '[ANONYMIZED]'])
            ->field('two_factor_secret')
                ->anonymizeWith('static_text', ['value' => '[ANONYMIZED]'])

            // Non-PII metadata: export only, never touched
            ->field('created_at')->exportable()
            ->field('locale')->exportable()
            ->field('orders_count')->exportable()

            ->retention(
                mode: RetentionMode::Delete,
                gracePeriodDays: 7, // opt in: 7 days to cancel; default would be 0
            )
            ->processOrder(1000); // subject is processed last
    }
}
```

### PII‑holder model — `Order`

```php
namespace App\Models;

use Illuminate\Database\Eloquent\Builder;
use Illuminate\Database\Eloquent\Model;
use GraystackIt\Gdpr\Contracts\PersonalData;
use GraystackIt\Gdpr\Enums\RetentionMode;
use GraystackIt\Gdpr\Support\PersonalDataBlueprint;
use GraystackIt\Gdpr\Traits\HasPersonalData;

class Order extends Model implements PersonalData
{
    use HasPersonalData;
    // Note: NO IsPersonalDataSubject — Order is not a subject

    public function personalData(PersonalDataBlueprint $b): PersonalDataBlueprint
    {
        return $b
            // PII: anonymize AND export
            ->field('shipping_address')
                ->anonymizeWith('address')
                ->exportable()
            ->field('billing_email')
                ->anonymizeWith('email')
                ->exportable()
            ->field('customer_notes')
                ->anonymizeWith('free_text')
                ->exportable()

            // Non-PII context for the export
            ->field('order_number')->exportable()
            ->field('total')->exportable()
            ->field('created_at')->exportable()

            ->retention(
                mode: RetentionMode::LegalHold,
                legalHoldDays: 3650,
                legalBasis: '§ 147 AO — tax record retention',
            )
            ->processOrder(100); // children before subject
    }

    public function scopePersonalDataForSubject(Builder $query, Model $subject): Builder
    {
        return match (true) {
            $subject instanceof User => $query->where('user_id', $subject->getKey()),
            default                  => $query->whereRaw('1 = 0'),
        };
    }
}
```

### Per‑field options

Both behaviors are **opt‑in**. A field has zero, one, or both of these. Zero is illegal.

| Method | Effect |
|---|---|
| `->anonymizeWith(string $alias, array $config = [])` | Mark field for anonymization. Resolved via `config('gdpr.anonymizers')`. |
| `->exportable()` | Mark field for inclusion in personal data exports. |

Combinations:

| Configuration | Anonymized? | Exported? | Use case |
|---|---|---|---|
| `->field('x')->anonymizeWith('y')->exportable()` | yes | yes | Standard PII (name, email, phone) |
| `->field('x')->anonymizeWith('y')` | yes | no | Internal PII (password hash, 2FA secret) |
| `->field('x')->exportable()` | no | yes | Non‑PII metadata (created_at, locale, computed counters) |
| `->field('x')` | no | no | ✗ throws on `build()` — field is functionless |

That is the **entire** per‑field API. Two flags total. No log flags — the audit log is event‑driven and never inspects fields.

### `->retention()` method signature

```php
public function retention(
    RetentionMode $mode = RetentionMode::Delete,
    int $gracePeriodDays = 0,
    ?int $legalHoldDays = null,
    ?string $legalBasis = null,
): self
```

Validation rules enforced on `build()`:

- `$gracePeriodDays >= 0` (default `0` means no grace, immediate processing)
- `$gracePeriodDays <= 30` → otherwise `InvalidRetentionConfig` exception
- If `$mode === RetentionMode::LegalHold` then `$legalHoldDays > 0` is required
- `$legalBasis` is optional regardless of mode (encouraged for `legal_hold`)

**`gracePeriodDays` only applies when the model is processed as a Subject** (i.e., the model uses `IsPersonalDataSubject` and someone called `requestDeletion()` on an instance). When the same model is processed as a Related model (registry walk after a different subject's grace expired), `gracePeriodDays` is ignored — by then we are past the grace window of the actual subject. Models without the `IsPersonalDataSubject` trait can omit `gracePeriodDays` entirely.

## 5. Deletion / Anonymization / Legal Hold Cycle

### Core principle

The grace period is implemented **entirely** in the `gdpr_deletions` table. **No row on the host model is touched during grace.** This means:

- No SoftDeletes requirement on host models
- No observer loops, no Eloquent lifecycle interference
- Cancellation is trivial (just update gdpr_deletions state)
- The host app controls auth/login behavior during grace via `GDPR::isDeletionPending()`

### State machine (per `gdpr_deletions` row)

```
                         ┌──────────────────────┐
                         │   ACTIVE / NORMAL    │
                         └──────────┬───────────┘
                                    │
                                    │ $user->requestDeletion()

                                    │  one for subject + one per affected model)

                  │       PENDING_GRACE                 │
                  │  (gdpr_deletions row exists,        │
                  │   host model row UNCHANGED)         │

                        │                │
            cancelDeletion              scheduled_for ≤ now()
                        │                │ OR deleteImmediately()
                        ▼                ▼
                ┌──────────────┐  ┌──────────────────────────┐
                │  CANCELLED   │  │  Snapshot.mode?          │
```

### State table

| State | Host row | gdpr_deletions row | Reversible? | Terminal? |
|---|---|---|---|---|
| `ACTIVE` | normal | none | n/a | no |
| `PENDING_GRACE` | unchanged | exists with `scheduled_for` | yes (`cancelDeletion`) | no |
| `CANCELLED` | unchanged | state=cancelled | n/a (already reverted) | yes |
| `ANONYMIZED` | fields wiped | state=anonymized | no | yes |
| `PENDING_LEGAL_HOLD` | fields wiped | state=pending_legal_hold, `hold_until` set | no | no |
| `ERASED` | row gone | state=erased | no | yes |

### Cron command `gdpr:process-deletions` (daily, idempotent)

```
Pass 1 — Grace expired:
  Group rows BY gdpr_request_id WHERE state = 'pending_grace' AND scheduled_for ≤ now()
  For each request:
    Sort rows by process_order ASC
    Dispatch as Bus::chain([Job(row1), Job(row2), ...])
  → Each chain runs sequentially: lower process_order first
  → If any job fails, the chain stops; remaining rows stay pending_grace
  → Next cron run picks them up again (idempotent)

Pass 2 — Legal hold expired:
  Group rows BY gdpr_request_id WHERE state = 'pending_legal_hold' AND hold_until ≤ now()
  For each request:
    Sort rows by process_order ASC
    Dispatch as Bus::chain([ForceDeleteJob(row1), ForceDeleteJob(row2), ...])
```

Both passes process per‑request, in `process_order` sequence, as chained jobs. Jobs re‑check state on entry to remain idempotent across retries.

### Cancellation (`cancelDeletion`)

Only valid while at least one `gdpr_deletions` row for this request is in `pending_grace`. Marks all such rows as `cancelled`, sets `gdpr_requests.status = cancelled`, fires `PersonalDataDeletionCancelled`. **There is nothing on host model rows to undo** because nothing was modified in the first place — that is the whole point of the design.

### Immediate deletion (`deleteImmediately`)

Skips remainder of grace by setting `scheduled_for = now()` on all `pending_grace` rows for this request and dispatching the cron pass immediately. No special code path.

### Auth during grace

The host model row is **unchanged** during grace, so by default the user can still log in. The host app decides what to do:

- **Google‑style** ("log in to cancel"): allow login, show banner, link to cancel
- **Facebook‑style** ("locked out"): block login

Provided helpers:

```php
GDPR::isDeletionPending($user); // bool

// Or as a query scope on Subject models (provided by IsPersonalDataSubject):
User::whereDeletionPending()->get();
User::whereNotDeletionPending()->get();

// Or as middleware on auth routes:
Route::middleware('gdpr.no-deletion-pending')->group(function () {
    Route::post('/login', LoginController::class);
});
```

### Domain events fired

| Event | When |
|---|---|
| `PersonalDataDeletionRequested` | `requestDeletion()` called |
| `PersonalDataDeletionCancelled` | `cancelDeletion()` called during grace |
| `PersonalDataAnonymized` | After fields wiped on a row (subject or related) |
| `PersonalDataErased` | After `delete()` on a row |
| `LegalHoldStarted` | A row enters `pending_legal_hold` |
| `LegalHoldExpired` | A row exits `pending_legal_hold` (deleted) |
| `PersonalDataExported` | Export job completed |

## 6. GDPR Compliance Notes

### Anonymization quality (README disclaimer)

The package ships an `anonymize` mode that replaces personal field values with non‑identifying placeholders. Whether the result is **legally** anonymization (out of GDPR scope) or **only** pseudonymization (still in scope) depends on **which fields the user marks as anonymizable**.

The README will include an "Anonymization done right" section warning about:

- **Quasi‑identifiers**: birthdate + ZIP + gender = 87 % unique (Sweeney 2000).
- **Behavioral patterns** that re‑identify by themselves (Netflix Prize attack).
- **Joinable timestamps + external datasets**.

The package gives the **tooling** for proper anonymization. Whether the user achieves it is their responsibility.

### Grace period guarantees

- **Default `0 days`** (no grace, immediate processing). The host app must opt in explicitly per profile by setting `gracePeriodDays`.
- **Hard cap at 30 days** — `PersonalDataBlueprint` throws on construction if higher (DSGVO Art. 12(3) "without undue delay, in any event within one month").
- User can always trigger `deleteImmediately()` to bypass remaining grace.
- User receives a confirmation mail via Notification when deletion is requested.
- During grace **nothing** is modified — cancellation is a complete clean rollback.
- With `gracePeriodDays = 0`, `requestDeletion()` and `deleteImmediately()` are equivalent: processing happens at the next cron pass (or synchronously if dispatched immediately).

### Audit log content rules

- ✅ Subject class + subject ID (FK; orphans after delete are correct)
- ✅ Event name from a fixed enum
- ✅ Timestamp
- ✅ `target_model` (nullable) for per‑model events
- ✅ `affected_rows` (nullable) row count
- ✅ Affected field **names** in `context` JSON (never values)
- ✅ Retention snapshot in `context` JSON for `deletion_scheduled` events
- ❌ No PII values
- ❌ No raw user agent
- ❌ Never an Eloquent observer triggered on save() — the audit log is event‑driven, written explicitly by the services that perform GDPR operations

After deletion, audit rows survive but their FK points nowhere → "we know an erasure happened on date X for subject Y, but Y no longer exists" → no re‑identification possible → exactly what GDPR wants.

### Legal hold

- Modeled via `mode = legal_hold` and `legal_hold_days`.
- The `legal_basis` field is **optional** in the profile and the user's responsibility. If the user enables legal hold, they should document why.
- The `AuditLogger` records `legal_basis` (when provided) into the `legal_hold_started` audit event.
- After `hold_until`, the row is deleted no matter what.

### Backups

The package cannot reach into backup files. The README will include a "Backups & GDPR" section reminding users to:

- Document backup retention in their privacy policy
- Re‑apply pending deletion requests after a restore
- Rotate backups within a documented window

### Subject↔Subject

When processing Subject A, Subject B is never touched, even if B has a foreign key to A. The host app must use:

- `->onDelete('set null')` on the FK migration, OR
- An event listener on `PersonalDataErased` to clean up references

The package will not silently null FKs across subjects — that's a host‑app concern.

### Auth bypass during grace

Because the package does not modify host rows during grace, a user could in principle keep using their account during grace and create new data — which would then be deleted at T+7. The README will document this and recommend wiring `GDPR::isDeletionPending()` or the `RequireNoDeletionPending` middleware into auth flows.

## Appendix B — Subject scope resolution

Each registered PII model defines `scopePersonalDataForSubject` to filter its rows for a given subject:

```php
public function scopePersonalDataForSubject(Builder $query, Model $subject): Builder
{
    return match (true) {
        $subject instanceof User     => $query->where('user_id', $subject->getKey()),
        $subject instanceof Customer => $query->where('customer_id', $subject->getKey()),
        default                      => $query->whereRaw('1 = 0'),
    };
}
```

For models the user cannot modify (vendor packages), the same scope can be defined externally and registered in the config:

```php
// config/gdpr.php
'models' => [
    \App\Models\User::class,
    \App\Models\Order::class,

    \Vendor\Package\ExternalModel::class => [
        'profile' => \App\Gdpr\Profiles\ExternalProfile::class,
        'scope'   => \App\Gdpr\Scopes\ExternalScope::class,
    ],
],
```

The `scope` class implements a single static method:

```php
final class ExternalScope
{
    public static function apply(Builder $query, Model $subject): Builder
    {
        // ...
    }
}
```

If neither method nor config scope is defined, the resolver logs a warning and skips the model. **Never an unscoped query.**

### Source resolution

| Field | Composer source | NPM source |
|---|---|---|
| `name` | `composer.lock` → `packages[].name` | `package-lock.json` → `packages` keys |
| `version` | `composer.lock` → `packages[].version` | `package-lock.json` → `packages[].version` |
| `description` | `composer.lock` → `packages[].description` | `node_modules/<pkg>/package.json` → `description` |
| `homepage` | `composer.lock` → `packages[].homepage` | `node_modules/<pkg>/package.json` → `homepage` (fallback: `repository.url`) |

**Composer**: All four fields are available directly in `composer.lock`. No extra file reads needed.

**NPM**: Lockfile only carries name + version. The scanner walks `node_modules/<pkg>/package.json` for each entry to enrich `description` and `homepage`. If `node_modules` is missing (e.g. CI without `npm install`), npm entries fall back to `name + version` only and `description`/`homepage` are `null` rather than throwing.

The scanner is resilient: missing fields anywhere become `null`, never errors. Field order in the output JSON is stable (alphabetical by name) to keep diffs clean across regenerations.

```json
"scripts": {
```

```php
Schema::create('gdpr_requests', function (Blueprint $table) {
    $table->id();

    // Subject (the one who initiated the request)
    $table->string('subject_type');
    $table->unsignedBigInteger('subject_id');
    $table->index(['subject_type', 'subject_id']);

    // Request metadata
    $table->string('type');   // 'delete' | 'anonymize' | 'export'
    $table->string('status'); // 'pending' | 'processing' | 'completed' | 'cancelled' | 'failed'
    $table->index('status');

    // Email snapshot at request time — used for notifications after subject
    // is anonymized/deleted. Wiped on final notification send (layer 1) or

    $table->string('notification_email')->nullable();

    // Export-specific: storage path for the generated file
    $table->string('export_file_path')->nullable();
    $table->timestamp('export_expires_at')->nullable();

    // Lifecycle timestamps
    $table->timestamp('requested_at');
    $table->timestamp('completed_at')->nullable();
    $table->timestamps();
});
```

The row itself is kept for 3 years (per Decision #24). The `notification_email` is wiped much sooner (per Decision #27). By the time the row is pruned, it contains no PII.

## Appendix G — `gdpr_deletions` schema

```php
Schema::create('gdpr_deletions', function (Blueprint $table) {
    $table->id();
    $table->foreignId('gdpr_request_id')->constrained();

    // Subject (the one who requested deletion)
    $table->string('subject_type');
    $table->unsignedBigInteger('subject_id');
    $table->index(['subject_type', 'subject_id']);

    // Affected model (one row per affected registered model)
    $table->string('target_model');
    $table->index('target_model');

    // Snapshot of the retention config at request time (NOT live)
    $table->json('retention_snapshot');

    // State machine
    $table->string('state'); // pending_grace | pending_legal_hold | anonymized | erased | cancelled
    $table->index('state');

    // Processing order (snapshot from profile, lower = earlier)
    $table->unsignedSmallInteger('process_order')->default(100);
    $table->index(['gdpr_request_id', 'process_order']);

    // Timing
    $table->timestamp('scheduled_for');         // grace expiry → Cron Pass 1
    $table->timestamp('hold_until')->nullable(); // legal hold expiry → Cron Pass 2
    $table->timestamp('processed_at')->nullable();

    $table->timestamps();
});
```

| `gdpr-views` | `resources/views/notifications/*.blade.php` → `resources/views/vendor/gdpr/notifications/*.blade.php` | no | Customize notification email branding (logo, footer, colours) without touching classes. |
| `gdpr-notifications` | `src/Notifications/*.php` → `app/Notifications/*.php` | no | Deep customization: add channels (Slack, SMS), attachments, conditional logic, CC recipients. |
| `gdpr` | all of the above | — | One‑shot initial setup when the user wants everything in their app. |

### Usage

```bash
# Minimal setup — required only
php artisan vendor:publish --tag=gdpr-config
php artisan vendor:publish --tag=gdpr-migrations
php artisan migrate

# Branding changes (common)
php artisan vendor:publish --tag=gdpr-views
php artisan vendor:publish --tag=gdpr-lang

# Deep customization (rare)
php artisan vendor:publish --tag=gdpr-notifications
# Then wire the classes in config/gdpr.php under 'notifications'

# Kitchen sink
php artisan vendor:publish --tag=gdpr
```

### Overriding published notifications

After publishing `gdpr-notifications`, the user must register their classes in `config/gdpr.php`:

```php
'notifications' => [
    // null     → use package default
    // false    → disable this notification
    // FQCN     → use custom class (must have the same constructor signature)
    'deletion_requested' => \App\Notifications\PersonalDataDeletionRequestedNotification::class,
    'deletion_cancelled' => null,
    'deletion_completed' => null,
    'export_ready'       => false,
],
```

Custom classes must accept the relevant request/event object as their single constructor argument to remain contract‑compatible with the `GdprManager` dispatch logic.

### Service Provider registration

```php
// In GdprServiceProvider::boot()
if ($this->app->runningInConsole()) {
    $this->publishes([
        __DIR__.'/../config/gdpr.php' => config_path('gdpr.php'),
    ], ['gdpr-config', 'gdpr']);

    $this->publishes([
        __DIR__.'/../database/migrations' => database_path('migrations'),
    ], ['gdpr-migrations', 'gdpr']);

    $this->publishes([
        __DIR__.'/../lang' => $this->app->langPath('vendor/gdpr'),
    ], ['gdpr-lang', 'gdpr']);

    $this->publishes([
        __DIR__.'/../resources/views' => resource_path('views/vendor/gdpr'),
    ], ['gdpr-views', 'gdpr']);

    $this->publishes([
        __DIR__.'/../src/Notifications' => app_path('Notifications'),
    ], ['gdpr-notifications', 'gdpr']);
}
```
